# C语言核心陷阱排雷笔记：浮点数、浅拷贝与指针

## 一、 浮点数精度丢失 (test3)

**现象**：将 `799.99` 赋值给 `float` 变量，`printf("%f")` 输出结果却是 `799.989990`。

**底层原因**：

1. **进制转换误差**：大部分十进制小数无法用二进制精确表示，只能截取近似值。
2. **`float` 精度限制**：遵循 IEEE 754 标准，单精度 `float`（4字节）只能保证 **6~7位** 有效十进制数字的精度。
3. **`printf` 默认行为**：`%f` 默认输出 6 位小数，导致底层的近似值（如 `799.989990234375`）被截断并显示出来。

**解决方案**：

* **用于显示**：通过 `%.2f` 限制输出格式，底层会自动四舍五入。
```c
printf("Price: %.2f\n", price); // 输出 799.99
```


* **用于运算**：将变量类型由 `float` 更改为双精度浮点数 `double`（提供 15~17 位有效数字），防止运算中误差累积。

## 二、 结构体赋值：浅拷贝与深拷贝 (test5)

**现象**：在 C 语言中，使用 `b = a;` 将一个结构体赋值给另一个结构体时，默认执行的是**浅拷贝（Shallow Copy）**。

**核心机制**：

* **浅拷贝（默认）**：C 语言进行逐字节的原样复制。对于普通数值（如 `int`, `float`），会复制具体的值；但对于**指针变量**，**只复制内存地址，不复制地址指向的实际数据**。
* **隐患**：
1. **数据共享/牵一发而动全身**：`a` 和 `b` 的指针成员指向同一块内存，通过其中一个指针修改数据，另一个也会受影响。
2. **野指针/内存崩溃**：如果释放了 `a` 中的指针内存（`free`），`b` 中的指针将变成指向未知区域的野指针。



**解决方案（深拷贝）**：
必须手动为新结构体的指针成员申请独立的内存空间，并复制真实数据。

```c
// 深拷贝示例
b.name = malloc(strlen(a.name) + 1); 
strcpy(b.name, a.name);              
```

## 三、 指针陷阱：重新赋值 vs 修改指向的内存 (test5)

**现象**：结构体浅拷贝后（`b = a`），执行 `a.name = "OH";`，发现 `b.name` 依然是 `"Toyota"`，并没有受到影响。

**底层原理解析**：
混淆了 **“改变指针的指向”** 和 **“改变内存里的真实内容”**。

1. **初始状态**：`a.name` 和 `b.name` 都存放了字符串字面量 `"Toyota"` 的首地址（例如 `0x1000`）。
2. 执行 `a.name = "OH";`：
* 编译器在内存（只读数据段）中创建了一个全新的字符串 `"OH"`（例如地址 `0x2000`）。
* 这行代码仅仅是把 `a.name` 变量里存放的地址改成了 `0x2000`。
* **关键点**：并没有去修改 `0x1000` 处的内容，因此 `b.name` 手里依然握着旧地址 `0x1000`，输出仍是 `"Toyota"`。



> **真正的浅拷贝“受害者”场景**：
> 如果执行的是 `a.name[0] = 'H';`（假设该内存可写），这才是**顺着地址去修改共享的内存内容**。此时因为 `b.name` 也看着这块内存，`b` 才会跟着改变。
